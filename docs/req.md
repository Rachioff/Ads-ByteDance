# AI广告推荐信息流 — 需求文档

> **版本**: v1.0  
> **作者**: zsy  
> **日期**: 2026-06-02

---

## 1. 项目概述

### 1.1 项目背景

在内容与广告混排的推荐场景中，用户是否愿意停留，取决于信息流的呈现质量和内容理解能力。本项目旨在构建一个**单列广告信息流 Android App**，在实现流畅的信息流浏览体验基础上，引入 **AI 大模型**为广告内容生成摘要与智能标签，并提供**对话式搜索**能力，让广告推荐更智能、更易被用户接受。

### 1.2 项目目标

| 目标维度 | 说明 |
|---------|------|
| **核心体验** | 实现单列广告信息流的流畅浏览、多样式卡片、频道切换、详情页交互、下拉刷新/上拉加载 |
| **AI 增强** | 大模型生成广告摘要与智能标签，支持标签过滤和自然语言对话式搜索 |
| **工程质量** | 模块化架构、资源复用机制、数据状态管理、埋点统计可视化 |
| **交付质量** | 代码可运行可复现、技术文档完整、演示视频清晰 |

### 1.3 范围界定

- **参赛形式**: 个人参赛，独立完成全部功能
- **平台**: Android（Kotlin）
- **功能覆盖**: 核心功能（全部）+ 可选 AI 功能（全部）
- **数据来源**: 本地 Mock 数据为主，AI 功能接入云端大模型 API

## 3. 核心功能需求

### 3.1 信息流浏览

#### 3.1.1 单列信息流布局

- **描述**: 采用单列垂直滚动列表展示广告内容，类似抖音/Reels 的单列沉浸式体验
- **功能点**:
  - 使用 RecyclerView 实现高性能列表
  - 支持流畅滚动（60fps，无卡顿）
  - 页面切换后返回时**保持列表滚动位置**
  - 支持 Fling 惯性滚动

#### 3.1.2 广告卡片多样式

- **描述**: 单列信息流中的每条广告以卡片形式呈现，至少支持 3 种样式
- **样式类型**:

| 卡片类型 | 布局特征 | 适用场景 |
|---------|---------|---------|
| **大图卡片** | 大尺寸封面图 + 标题 + 摘要 + 广告主信息 + 互动按钮 | 品牌广告、高质量视觉内容 |
| **小图卡片** | 左文右图/左图右文 + 标题 + 标签 + 互动按钮 | 电商商品、应用推荐 |
| **视频卡片** | 视频播放区 + 标题 + 摘要 + 广告主信息 + 互动按钮 | 视频广告、产品演示 |

- **统一元素**: 所有卡片均包含：点赞按钮、收藏按钮、分享按钮、曝光/点击计数展示区

#### 3.1.3 频道切换

- **描述**: 顶部 Tab 栏支持切换不同内容频道，切换时刷新数据
- **频道列表**:
  - **精选**（默认）— 推荐算法排序的综合广告流
  - **电商** — 电商/购物类广告
  - **本地** — 本地生活服务类广告
- **交互要求**:
  - Tab 选中态有视觉高亮和切换动画
  - 切换 Tab 时列表回到顶部并刷新数据
  - 支持左右滑动切换 Tab（ViewPager2 + TabLayout）

#### 3.1.4 下拉刷新与上拉加载

- **下拉刷新**:
  - 列表顶部下拉触发数据刷新
  - 刷新时显示加载动画（Material Design 风格）
  - 刷新完成后回到列表顶部
- **上拉加载更多**:
  - 滚动到底部自动触发分页加载
  - 加载中显示底部加载指示器
  - 全部加载完毕显示"没有更多了"
- **状态处理**:
  - **加载中**: 显示 loading 动画
  - **空态**: 显示空数据提示（含图标 + 文案）
  - **错误态**: 显示错误提示 + 重试按钮
  - **网络异常**: 显示网络错误提示

### 3.2 广告详情页

#### 3.2.1 详情页展示

- **描述**: 点击信息流中的广告卡片进入详情页
- **详情页类型**:
  - **图文详情**: 大图/多图轮播 + 完整文案 + 广告主信息
  - **视频详情**: 视频播放器（内流自动播放） + 完整文案 + 广告主信息

#### 3.2.2 页面导航与位置保持

- **进入**: 从信息流点击卡片 → 过渡动画 → 详情页
- **返回**: 按返回键/返回按钮 → 退出动画 → 回到信息流**原位置**
- **实现约束**: 使用 Activity/Fragment 导航，确保 onResume 时恢复滚动位置

#### 3.2.3 跨页面状态同步

- **描述**: 详情页和信息流页面的互动状态（点赞/收藏/分享）保持同步
- **同步场景**:
  - 用户在详情页点赞 → 返回信息流，对应卡片点赞状态已更新
  - 用户在详情页收藏 → 返回信息流，对应卡片收藏状态已更新
- **数据流**: 本地状态管理（ViewModel/Repository） → UI 观察更新

### 3.3 资源复用机制

| 复用类型 | 描述 | 要求 |
|---------|------|------|
| **Cell 复用** | RecyclerView ViewHolder 复用 | 避免重复 inflate，减少内存分配 |
| **播放器复用** | 视频播放器实例池化管理 | 同一时刻最多 1 个活跃播放器（外流），进入内流时复用 |
| **缓存池** | 图片/视频资源的 LRU 缓存 | 内存缓存 + 磁盘缓存二级策略 |

### 3.4 数据获取与管理

#### 3.4.1 数据获取方式

- **主模式**: 本地 Mock 数据（JSON 文件内置）
- **扩展模式**: 网络请求（OkHttp + Retrofit），支持切换到真实 API
- **切换机制**: 通过配置开关（BuildConfig 字段）在 Mock 和网络模式间切换

#### 3.4.2 数据模型

- **广告数据结构**:
  - 广告 ID、标题、描述、封面图 URL/资源、视频 URL/资源
  - 广告主名称、头像
  - 广告类型（大图/小图/视频）
  - 所属频道（精选/电商/本地）
  - 标签列表（品类、风格、受众、场景）
  - AI 摘要（可选，由大模型生成）
  - 互动数据（点赞数、收藏数、分享数）
  - 曝光次数、点击次数

#### 3.4.3 数据生命周期管理

- **加载**: 首次进入 → 加载首页数据 → 缓存到内存
- **刷新**: 下拉刷新 → 清除当前页缓存 → 重新加载第一页
- **翻页**: 上拉加载 → 追加下一页数据 → 合并到内存缓存
- **销毁**: 页面销毁 → 释放不必要的内存缓存（保留磁盘缓存）

### 3.5 视频播放器

#### 3.5.1 播放控制

| 控制项 | 外流（信息流中） | 内流（详情页） |
|--------|-----------------|---------------|
| **默认状态** | 暂停（显示封面图 + 播放按钮） | 自动播放 |
| **播放/暂停** | 点击播放按钮开始播放 | 点击视频区域暂停/继续 |
| **静音** | 默认静音，点击音量按钮取消静音 | 默认有声，可切换静音 |
| **进度** | 显示播放进度条 | 显示完整播放控制条 |

#### 3.5.2 播放器资源复用

- 同一时刻信息流中最多 1 个视频处于播放状态
- 滚动到下一个视频卡片时，释放上一个播放器资源
- 进入视频详情页时复用当前播放器实例

### 3.6 动画效果

| 动画类型 | 描述 | 触发场景 |
|---------|------|---------|
| **页面切换** | 信息流 → 详情页的共享元素过渡动画 | 点击卡片进入详情 |
| **点赞动效** | 点赞时播放彩蛋动画（如爱心扩散、撒花） | 点击点赞按钮 |
| **Tab 切换** | Tab 选中指示器平滑滑动 | 切换频道 |
| **刷新动画** | 下拉刷新时的 Material 加载动效 | 下拉刷新 |
| **按钮反馈** | 收藏/分享按钮的缩放或颜色动效 | 点击收藏/分享 |

### 3.7 埋点统计

#### 3.7.1 曝光统计

- **曝光定义**: 广告卡片进入屏幕可视区域 ≥ 50% 且停留 ≥ 1 秒
- **统计方式**: 本地计数器 + 页面展示
- **去重**: 同一广告在同一次信息流会话中只计 1 次曝光

#### 3.7.2 点击统计

- **点击定义**: 用户点击广告卡片进入详情页
- **统计方式**: 本地计数器 + 页面展示

#### 3.7.3 统计可视化

- **展示位置**: 独立统计页面或信息流顶部统计入口
- **展示内容**:
  - 每个广告的曝光次数、点击次数
  - 点击率（CTR = 点击数 / 曝光数）
  - 列表形式展示，支持排序

#### 3.7.4 互动模拟

- **点赞**: 本地状态管理，支持点赞/取消点赞，更新计数
- **收藏**: 本地状态管理，支持收藏/取消收藏
- **分享**: 模拟分享行为，弹出分享面板（调用系统分享 Sheet）

### 3.8 用户行为统计与个性化推荐

#### 3.8.1 功能概述

在广告维度的埋点统计（3.7 节）基础上，引入**用户维度**的行为追踪。通过采集用户对广告的各类互动行为，构建用户兴趣画像，驱动**个性化推荐排序**——不同兴趣偏好的用户将在精选频道看到不同的广告排序结果。

#### 3.8.2 用户行为采集

| 行为类型 | 触发条件 | 权重 | 记录内容 |
|---------|---------|------|---------|
| **点击** | 用户点击广告卡片进入详情页 | 1 | 广告 ID、涉及的标签列表、时间戳 |
| **点赞** | 用户点击点赞按钮 | 2 | 广告 ID、涉及的标签列表、时间戳 |
| **收藏** | 用户点击收藏按钮 | 3 | 广告 ID、涉及的标签列表、时间戳 |
| **分享** | 用户触发分享行为 | 2 | 广告 ID、涉及的标签列表、时间戳 |
| **标签点击** | 用户点击卡片上的标签 Chip | 1 | 被点击的标签名称、时间戳 |
| **搜索** | 用户提交对话式搜索查询 | 1 | 搜索关键词/意图标签、时间戳 |

- **存储方式**: 本地持久化存储，确保行为数据不丢失

#### 3.8.3 用户兴趣画像

- **画像构建**: 汇总用户所有历史行为，按标签维度聚合权重得分

  ```
  标签偏好得分 = Σ(该标签下各行为次数 × 对应行为权重)
  ```

  例如：用户点击了 3 条带"运动"标签的广告、点赞了 2 条、收藏了 1 条
  → "运动"标签偏好得分 = 3×1 + 2×2 + 1×3 = 10

- **画像更新**: 每次用户产生新行为后实时更新
- **冷启动**: 新用户无行为数据时使用默认排序，随行为累积逐步形成画像

#### 3.8.4 推荐策略差异化

- **精选频道排序**: 根据用户兴趣画像对广告进行个性化排序

  | 用户状态 | 排序策略 |
  |---------|---------|
  | **新用户**（无行为数据） | 默认排序（按广告热度/发布时间降序） |
  | **有行为数据用户** | 按广告标签与用户偏好标签的**匹配度**降序排列 |

- **匹配度计算**:

  ```
  匹配度 = Σ(广告每个标签在用户画像中的权重得分)
  ```

  若广告标签为 ["运动", "学生党", "性价比"]，用户在"运动"=10、"学生党"=5、"性价比"=2，则匹配度 = 17。匹配度越高，排序越靠前。

- **其他频道**: 电商/本地频道维持默认排序（频道本身已是内容分类），可选也加入偏好权重

#### 3.8.5 偏好可视化

- **展示位置**: 在统计页面新增"我的偏好"区域
- **展示内容**:
  - 用户 Top 5-10 偏好标签及权重得分
  - 标签云/柱状图等可视化形式
  - 行为数据总览（总点击次数、总点赞数、总收藏数等）
- **效果可感知**: 用户在精选频道可直观看到偏好标签相关的广告排在更前面

## 4. 可选功能需求（AI 增强）

### 4.1 AI 摘要与智能标签生成

#### 4.1.1 功能描述

调用云端大模型 API，为每条广告自动生成：
- **摘要**（1-2 句话）：用于卡片副标题或详情页摘要区
- **智能标签**（3-5 个）：涵盖品类、风格、受众、场景等维度

#### 4.1.2 标签维度

| 标签类别 | 示例值 | 说明 |
|---------|--------|------|
| **品类** | 美妆、数码、服饰、食品、家居 | 商品/服务所属品类 |
| **风格** | 简约、潮流、复古、轻奢 | 视觉/品牌风格 |
| **受众** | 学生党、职场人、宝妈、运动爱好者 | 目标用户群体 |
| **场景** | 通勤、居家、户外、聚会 | 使用/消费场景 |

#### 4.1.3 大模型接入要求

- **模型选择**: Qwen（通义千问）免费版，或 OpenAI 兼容 API
- **调用方式**: 客户端 → HTTP POST → 模型 API → 返回 JSON
- **输入**: 广告标题 + 描述 + 广告主信息
- **输出约束**: 结构化 JSON（摘要字段 + 标签数组），通过 Prompt 约束输出格式
- **缓存策略**: 同一条广告的 AI 结果缓存到本地，避免重复调用
- **降级方案**: 若模型不可用，使用预设的静态摘要和标签

#### 4.1.4 展示位置

- **信息流卡片**: 摘要显示在标题下方作为副标题；标签以小标签（Chip）形式展示在卡片底部
- **详情页**: 摘要显示在顶部摘要区；标签以 FlowLayout 展示

### 4.2 标签点击过滤

#### 4.2.1 功能描述

用户点击信息流卡片上的某个标签后，当前频道下仅展示包含该标签的广告。

#### 4.2.2 交互流程

1. 用户点击卡片上的标签 Chip
2. 顶部出现过滤状态栏（显示当前过滤标签 + 清除按钮）
3. 列表刷新，仅展示包含该标签的广告
4. 用户点击清除或切换 Tab 后取消过滤

#### 4.2.3 数据流

- 本地过滤（不发起网络请求），基于已加载数据按标签匹配
- 过滤后列表支持下拉刷新和上拉加载（带过滤条件的 Mock 查询）

### 4.3 对话式搜索

#### 4.3.1 功能描述

提供聊天式搜索界面，用户以自然语言描述想看的广告类型，系统返回匹配的广告结果。

#### 4.3.2 交互流程

1. 用户从信息流页面进入搜索入口（如顶部搜索图标或底部导航）
2. 进入聊天式界面，显示欢迎语和输入框
3. 用户输入自然语言查询（如"推荐一些适合学生党的平价数码产品"）
4. 系统解析查询意图 → 匹配广告 → 以卡片列表形式返回结果
5. 支持多轮对话（上下文保持）

#### 4.3.3 技术实现方案

- **意图解析**: 调用大模型 API 将自然语言转为结构化搜索条件（品类、受众、风格、场景、价格区间等）
- **匹配引擎**: 将搜索条件与本地广告数据匹配（标签匹配 + 关键词匹配）
- **结果排序**: 按匹配度降序排列
- **降级方案**: 大模型不可用时退化为关键词搜索

#### 4.3.4 UI 要求

- 聊天气泡式界面（用户消息右对齐，系统回复左对齐）
- 搜索结果以广告卡片形式嵌入对话流
- 点击搜索结果卡片可进入详情页
- 支持清空对话、重新开始

### 4.4 图片/视频加载与缓存

#### 4.4.1 图片加载策略

| 策略 | 描述 |
|------|------|
| **首屏优先** | 优先加载屏幕可见区域内的图片，非可见区域延迟加载 |
| **滚动复用** | 复用 RecyclerView 的 ImageView，取消离开屏幕的图片请求 |
| **渐进加载** | 先显示低分辨率缩略图 → 再替换为高清图 |
| **失败重试** | 加载失败自动重试 3 次，仍失败则显示占位图 |
| **磁盘缓存** | 已加载图片缓存到磁盘，下次冷启动可直接展示 |

#### 4.4.2 视频加载策略

| 策略 | 描述 |
|------|------|
| **预加载** | 当前视频播放时，预加载下一个视频的首帧/封面 |
| **封面图** | 视频未播放时显示封面图，加载成功后显示封面 |
| **智能预判** | 用户快速滑动时不加载视频，停止滑动后才加载当前可见视频 |

#### 4.4.3 缓存架构

- **内存缓存**: LRU Cache（图片默认 1/8 可用内存）
- **磁盘缓存**: DiskLRU Cache（默认 200MB，支持清理）
- **缓存键**: URL 的 MD5 哈希值
- **缓存过期**: 基于 HTTP Cache-Control 头或默认 7 天

## 5. 非功能性需求

### 5.1 性能

| 指标 | 目标值 | 测量方式 |
|------|--------|---------|
| 列表滚动帧率 | ≥ 55fps | Android Profiler GPU 渲染 |
| 首屏加载时间 | < 2s（冷启动） | 计时 |
| 图片加载时间 | < 500ms（单张） | 网络模拟 |
| 内存占用 | < 200MB（正常运行） | Memory Profiler |
| APK 体积 | < 30MB | 构建产物 |

### 5.2 稳定性

- **无 ANR**: 所有耗时操作在子线程执行，主线程操作 < 5s
- **无内存泄漏**: Activity/Fragment 销毁后正确释放资源（LeakCanary 验证）
- **崩溃处理**: 全局异常捕获，Crash 率目标 0%

### 5.3 可用性

- **空态处理**: 无数据时展示友好的空态页面
- **加载态**: 所有异步操作有 loading 指示
- **错误态**: 网络错误/数据异常有明确提示和重试入口
- **边界处理**: 网络超时、数据为空、图片加载失败等均有兜底方案

### 5.4 可维护性

- **模块化**: 按功能拆分模块，模块间通过接口通信
- **代码规范**: 统一代码风格，关键逻辑有注释
